Evaluation Plan Dalam MySQL

Pada awalnya, query dapat dieksekusi dengan baik lancar, namun seiring dengan berjalannya waktu, eksekusi query terasa lambat entah karena query yang tidak optimal atau data yang semakin besar. Jika sudah demikian, maka mau tidak mau kita harus melakukan optimasi query MySQL. Kenapa? karena hal ini akan berdampak signifikan bagi kelangsungan aplikasi kita.

Salah satu cara yang bisa digunakan untuk optimasi query adalah dengan melihat Evaluation plan dari database kita. Evaluation plan adalah dokumen tertulis yang menjelaskan bagaimana seorang programmer akan melihat dan mengevaluasi programnya, serta bagaimana programmer menggunakan hasil dari evaluasi untuk memperbaiki program yang di buat dan pengambilan keputusan terhadap program tersebut. Sebagian besar sistem database menyediakan cara untuk melihat evaluation plan yang dipilih untuk dijalankan queri tertentu. Biasanya yang terbaik adalah menggunakan GUI yang disediakan dengan sistem database untuk melihat rencana evaluasi. Namun, jika Anda menggunakan commandline interface, banyak database mendukung variasi dari perintah Explain < query > yang menampilkan execution plan yang dipilih untuk queri yang ditentukan. Pada MySQL cara menampilkan evaluation plan menggunakan syntax EXPLAIN.

Penjelasan dari tabel di atas.

1. id

    Kolom id berisi identifier dari tabel, Pada contoh di atas keduanya bernilai 1.

2. select_type

    Kolom select_type menunjukkan tipe select yang digunakan. Pada contoh di atas keduanya simple, ini berarti bahwa tipe select yang digunakan keduanya adalah tipe select sederhana.

3. table

    Kolom table menunjukkan inisalisasi nama dari tabel yang kita gunakan pada code kita, Tabel pertama berisi m yang merupakan inisalisasi nama dari tabel mahasiswa, sedangkan tabel dua isinya p yang meruoakan inisalisasi nama dari tabel penerima_beasiswa.

4. type

    Kolom type berisi informasi tentang join type yang digunakan. Pada hasil diatas terdapat dua tipe yaitu tipe ALL dan ref. Untuk tipe ALL adalah Pemindaian tabel lengkap dilakukan untuk setiap kombinasi baris dari tabel sebelumnya. Untuk tipe ref adalah Semua baris dengan nilai indeks yang cocok dibaca dari tabel ini untuk setiap kombinasi baris dari tabel sebelumnya. Ref digunakan jika gabungan (join) hanya menggunakan awalan leftmost dari key atau jika key bukan indeks PRIMARY KEY atau UNIK (dengan kata lain, jika bergabung tidak dapat memilih baris tunggal berdasarkan nilai kunci). Jika kunci yang digunakan hanya cocok dengan beberapa baris, ini adalah tipe gabungan yang baik.

5. possible_keys

    Kolom possible_key menunjukkan key dalam tabel yang bisa digunakan untuk menemukan baris dalam tabel. Hasil di atas menujukkan null dan primary.

6. key

    Kolom possible_key menunjukkan nama key yang digunakan untuk mengambil baris. Pada contoh di atas menujukkan null dan primary.

7. key_len

    Kolom key_len menunjukkan panjang key yang digunakan MySQL. Pada hasil di atas baris pertama bernilai Null karena tidak ada key yang digunakan sedangkan baris kedua bernilai 42

8. ref

    Kolom ref menunjukkan nama kolom dari tabel yang diproses dengan key. Pada hasil eksekusi di atas baris pertama bernilai Null karena tidak ada yang diproses dengan key sedangkan baris kedua menunjukkan kolom NIM pada table m dan database tgsbasdatakhir.

9. row

    Kolom row menunjukkan jumlah baris yang di temukan ketika code di atas dijalankan. Pada hasil di atas tabel m memilki jumlah row 3 dan tabel p memiliki jumlah row 1.

10. Extra

    Kolom Extra menunjukkan informasi tambahan tentang bagaimana MySQL mengeksekusi query. Pada hasil di atas tabel m tidak ada isinya dan pada tabel p terdapat using where, using index, dan not exist. using where berarti bahwa dalam query yang kita buat kita menggunakan where (WHERE p.nim IS NULL). using index berarti bahwa untuk mengambil atau memperoleh informasi dari tabel p, kita hanya menggunakan index. Not exist berarti bahwa hentikan untuk mencari baris lain jika sudah memperoleh baris yang tepat
